commitgpt: The Git Hook I Built Because I Was Tired of Writing Commit Messages

2025Python · CLI · Developer Tools

Conventional commits are the right way to write commit messages. Structured prefixes — feat, fix, chore, refactor, docs — make history readable, power changelog generation, and enforce a discipline that prevents the project-ending commit message of 'stuff' appearing at a critical moment.

They're also tedious to write. Not hard. Just tedious. You make a change, you know what the change is, and then you have to translate that into the right prefix and a clear one-line description. Every time. For every commit. I was doing it, but I wasn't enjoying it.

How It Works

commitgpt hooks into Git's commit-msg flow. When you run git commit, Git executes the commit-msg hook before finalizing the commit. commitgpt reads the staged diff, sends it to an LLM with a structured prompt, and writes a conventional commit message back to the commit file.

You never type a commit message. You just commit. The message appears.

Why a Hook, Not a CLI

A standalone CLI would mean adding a step: run the CLI, copy the message, paste it into git commit. That's not meaningfully less tedious than writing the message yourself.

The hook integrates into the existing workflow. You don't change how you commit. The tool disappears into the background. That's the goal — tools should be invisible.

Design Decisions

Conventional commits specifically because they're the most widely adopted structured commit standard and because the structure (type, optional scope, description) maps cleanly to what an LLM can infer from a diff. 'This diff adds a new function to the auth module' → feat(auth): add token refresh function.

The LLM is sometimes wrong. It occasionally picks the wrong type, or writes a description that's technically accurate but misses the intent. There's an escape hatch: if you don't like the generated message, you edit the commit file before the hook finalizes. Standard Git behavior, no special commands.

The prompt is the engineering. Getting the LLM to consistently output valid conventional commit format — and nothing else — took more iteration than the hook integration itself.

The Philosophy

Developer tools should disappear. The best tools are the ones you stop noticing because they just work. You shouldn't think about commitgpt while you're committing — you should just commit, and the message should be there.

I have 56 public repos. I use commitgpt on all of them now.
